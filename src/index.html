Radix Sort

Hoje, temos a missão de destrinchar e perscrutar, no intuito de sermos comparáveis (em bom tom) com Bentinho, algo que nem Machado de Assis conseguiria imaginar que um dia existiria. Sim, ele mesmo, o magnânimo Radix Sort, com uma lógica bastante clara e simplória, mas com uma implementação pra lá de oblíqua e dissimulada à primeira vista. Sabe como é né, tipo os olhos de Capitu.

Exilar-nos-emos dessa confusão, a contragosto de Gonçalves Dias. Vamos lá!

Introdução

O Radix Sort está na prateleira dos algoritmos lineares, ou seja, como diria o filósofo contemporâneo nacional mais importante e que joga com os urubus da Gávea, é "oto patamá". Mas calma lá, aprendiz de Dom Quixote! Vamos deixar nosso mundo menos cortês e cor de rosa.

O modelo de ordenação do Radix é organizar os números processando seus dígitos, e isso tende a ser do menos significativo para o mais significativo (Ok, senhor do contra, pode ser ao contrário também). Ele utiliza um algoritmo de ordenação estável, ou seja, mantém a ordem relativa dos elementos com os mesmos valores igual no estado inicial desordenado. Utilizaremos aqui o Counting Sort, para ordenar os elementos por buckets baseados em cada "dígito"!

Complexidade

Só que antes de partir para o código, falemos de complexidade, pois é verdade que ele é linear, porém nem sempre.

Seja:
n: o número de elementos no array.
k: o número máximo de dígitos dos números.

Então:
1. Para cada dígito (k), temos que fazer uma ordenação estável em todo o array.
2. Se contarmos apenas com o Counting Sort, ele tem uma complexidade de O(n + r), no qual n é o número de elementos e r o intervalo dos dígitos, que normalmente é 10, por conta do sistema decimal.

Logo, a complexidade total do Radix Sort é O(k * (n + r))!

É importante notar que se os números forem negativos, seu algoritmo não funcionará, visto que não não possuem uma representação de dígitos separada que pode ser ordenada diretamente nesse contexto. Já o número zero, é tratado normalmente.

Simplificando a complexidade total, temos que se k e r forem considerados constantes, então pode se esbaldar com a sua mania de matemático açougueiro e passar a faca nas constantes, deixando nossa complexidade em O(n).

Mas quando o Radix Sort não é linear? Por exemplo, quando k não for constante ao ter muitos dígitos ou então o intervalor r for muito grande, crescendo a complexidade.

Em suma:
1. Sim, é O(n) em casos práticos onde k e r são pequenos ou constantes, sendo mais eficiente.
2. O(k * (n + r)) é o caso geral, mas no pior caso, pode até significativamente ineficiente se comparado com nosso querido Roma Aeterna, que divide para conquistar e é O(nlogn), o MergeSort.

Métodos Privados


    private int getMax(int arr[], int n) {
        int maior = arr[0];
    
        for (int i = 1; i < n; i++) {
            if (arr[i] > maior) maior = arr[i];
        }
        return maior;
    }

Para que nosso algoritmo funcione, antes precisamos declarar alguns métodos privados para nosso auxílio. O primeiro deles é o getMax, cujo objetivo nada mais e nada menos é que obter o maior elemento de um array de inteiros.

Na primeira linha, a variável maior é inicializada com o primeiro elemento (arr[0]). O próximo passo é fazer um laço para percorrer o array e comparar o nosso atual maior elemento com os elementos restantes. Nosso laço começa em i = 1, justamente pois maior já foi inicializado como primeiro elemento, continuando até que i seja menor que n, verificando todos os elementos até o índice n - 1. Por fim, mas não menos importante, retornamos a nossa variável com o maior elemento.


    private void countSort(int arr[], int n, int exp) {
        int output[] = new int[n];
        int i;
        int count[] = new int[10];
        Arrays.fill(count, 0);
    
        for (i = 0; i < n; i++)
            count[(arr[i] / exp) % 10]++;
    
        for (i = 1; i < 10; i++)
            count[i] += count[i - 1];
    
        for (i = n - 1; i >= 0; i--) {
            output[count[(arr[i] / exp) % 10] - 1] = arr[i];
            count[(arr[i] / exp) % 10]--;
        }
    
        for (i = 0; i < n; i++)
            arr[i] = output[i];
    }

Agora, respeitável público, implementaremos o Counting Sort para ordernar os elementos por buckets e dígitos. Note desde já, que lidaremos com um algoritmo modificado, justamente para abarcar nossa demanda por ordenação através de dígitos unitários.

Nas primeiras quatro linhas, estaremos lavrando a terra com a inicialização do que necessitaremos. Primeiro, declaramos um array temporário chamado output, no qual armazenaremos nossa saída ordenada. Por simplicidade de não ter que ficar declarando o índice (i) dos laços toda vez, já o fiz no início. Em seguida, um array chamado count de tamanho 10 (para dígitos de 0 a 9) que será usado para contar a ocorrência dos dígitos no dígito atual (exp), ademais também inicializei todas as posições do array com 0, utilizando o método Arrays.fill.

Partindo para o primeiro laço, percorreremos todo o nosso array recebido de argumento com os nossos elementos, e a cada elemento, o dígito correspondente no dígito atual (exp) é calculado como (arr[i] / exp) % 10. Logo após, incrementaremos a frequência desse dígito no array count.

No segundo laço, iremos converter count em um array acumulado. Agora, count[i] representa a posição final de um número com o dígito i no array ordenado.

O terceiro laço terá uma dinâmica diferente, percorremos o array de trás para frente, pois queremos manter a estabilidade da ordenação, ao contrário, não estariamos utilizando o Counting. Na primeira linha, encontramos a posição correta de arr[i] no array output usando o valor acumulado em count e então colocamos arr[i] na posição calculada. Por último, decrementamos o valor correspondente no array count.

O último laço, nada mais faz que copiar o nosso resultado ordenado do array output para o array original, arr.

Método Público


    public void radixSort(int arr[], int n) {
        int maior = getMax(arr, n);
    
        for (int exp = 1; maior / exp > 0; exp *= 10)
            countSort(arr, n, exp);
    }

Finalmente, a parte mais simples e vital, o nosso algoritmo verdadeiro. Iniciamos procurando pelo maior com o método privado que definimos anteriormente, pois o maior valor determina o número de dígitos necessários para realizar as iterações do Radix Sort. Por exemplo, se o maior número for 123, será necessário processar três dígitos (unidades, dezenas e centenas).

Logo então, declaramos um laço para iterar sobre cada posição decimal. Para representar as unidades, exp começa como 1. A cada iteração, o exp é multiplicado por 10, virando dezena, centena e assim por diante. O laço termina quando maior / exp > 0, ou seja, quando exp ultrapassa o número de dígitos do maior elemento. Tomando novamente como exemplo o 123, o laço irá iterar para exp = 1 (unidades), exp = 10 (dezenas), e exp = 100 (centenas).
